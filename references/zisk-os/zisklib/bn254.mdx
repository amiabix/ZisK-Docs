<Summary>

`zisklib` functions for BN254 field arithmetic (Fp, Fp2, Fr), G1
curve operations, and pairing computations.

</Summary>

## Module overview

BN254 (a.k.a. alt_bn128) is the pairing-friendly curve exposed by
the Ethereum precompiles in EIP-196 and EIP-197, and the curve most
SNARK verifiers target on L1: Groth16 verifiers and PLONK verifiers
both reduce to a small number of pairing checks against BN254 G1
and G2 inputs.

The pairing check is the verifier's core operation. Everything else
on this page (Fp, Fp2, Fr, G1) feeds it: scalar multiplication
forms verifier inputs, G1 / G2 carry the proof and verification
key, and the pairing maps them into Fp12 where equality is decided.

The type hierarchy:

- **Fp** is the base field, integers mod the prime `p` (~254 bits).
  G1 coordinates live here.
- **Fp2** is the quadratic extension over Fp. G2 coordinates live
  here, and the pairing target Fp12 is built on top of it.
- **Fr** is the scalar field, integers mod the subgroup order `r`.
  Used for scalar multiplication on G1 and G2.
- **G1** is the curve `y² = x³ + 3` over Fp.
- **G2** is the twisted curve over Fp2.
- **Pairing** is the bilinear map `e: G1 × G2 → Fp12`. The Miller
  loop computes the rational accumulator; final exponentiation
  projects it into the target subgroup so equality testing is
  meaningful.

## When to use BN254 vs BLS12-381

Use BN254 when the consumer is an Ethereum L1 precompile or a
SNARK verifier built on Groth16 or PLONK that targets those
precompiles. Use [BLS12-381](/references/zisk-os/zisklib/bls12-381)
for BLS aggregate signatures, KZG commitments, and modern proof
systems where the larger 381-bit field gives stronger security
margins at the cost of per-operation work.

:::note
G1 points are `[u64; 8]` (x ++ y, 4 LE limbs per coordinate, 256 bits each).
G2 points are `[u64; 16]` (Fp2 x ++ Fp2 y, 4 LE limbs per Fp element).
GT elements (Fp12) are `[u64; 48]`.
:::

:::note hints feature gate
Every BN254 routine that calls a syscall takes an extra
`#[cfg(feature = "hints")] hints: &mut Vec<u64>` parameter. The
argument is only present when the `hints` Cargo feature is enabled,
so callers built without that feature pass nothing. The signatures
below show the parameter exactly as it appears in source; omit it
(and the trailing `#[cfg(feature = "hints")] hints,` argument at
call sites) when the feature is off.
:::

## Base field (Fp)

Arithmetic mod the BN254 base prime `p`. Fp elements are the raw
coordinates of G1 points and the components of every Fp2 element,
so Fp operations underlie every routine in the rest of the page.

## Extension field (Fp2)

The quadratic extension Fp2 = Fp[i] / (i² + 1). Its elements are
pairs (re, im) of Fp values. G2 coordinates are Fp2 elements, and
the pairing target Fp12 is built as a tower over Fp2, so these
routines are the bridge from Fp into the pairing machinery.

## Pairing

The bilinear map `e: G1 × G2 → Fp12`. The Miller loop accumulates
line evaluations along a chain of G2 doublings and additions, and
final exponentiation lifts the result into the order-`r` subgroup
of Fp12 where pairing equality is well-defined. SNARK verifiers
typically evaluate `pairing_check_bn254` on a small fixed-size
batch.
