<Summary>

`zisklib` functions for BLS12-381 field arithmetic (Fp, Fp2, Fr),
G1 and G2 curve operations, pairing computations, KZG verification,
and hash-to-curve helpers.

</Summary>

## Module overview

BLS12-381 is the pairing-friendly curve behind Ethereum BLS
aggregate signatures (EIP-2537), Filecoin proofs, Zcash Sapling,
the validator signatures that secure Ethereum consensus, and the
KZG commitments used by the Ethereum blob market (EIP-4844).

Its base field is ~381 bits wide compared to BN254's ~254 bits, so
each field op and pairing is more expensive, but the curve
delivers a stable ~128-bit security target after recent advances
in finite-field discrete log attacks ruled BN254 below that bar.

The type hierarchy:

- **Fp** is the base field, integers mod the prime `p` (~381 bits).
  G1 coordinates live here.
- **Fp2** is the quadratic extension. G2 coordinates live here.
- **Fp6** is the cubic extension over Fp2; **Fp12** is the
  quadratic extension over Fp6 and the pairing target.
- **Fr** is the scalar field, integers mod the subgroup order `r`.
  Used for scalar multiplication on G1 and G2.
- **G1** is the curve `y² = x³ + 4` over Fp.
- **G2** is the twisted curve over Fp2.
- **Pairing** is the bilinear map `e: G1 × G2 → Fp12`. The Miller
  loop accumulates line evaluations and final exponentiation
  projects the result into the order-`r` subgroup of Fp12, the
  cyclotomic subgroup `GΦ₆(p²)`, where pairing equality is
  decided.
- **KZG** is a polynomial commitment scheme built on the pairing.
  An opening of a polynomial `f` at point `z` to value `y` is
  verified with one pairing equation in BLS12-381; this is the
  primitive Ethereum blobs use.

:::note
G1 points are `[u64; 12]` (x ++ y, 6 LE limbs per coordinate, 384 bits each).
G2 points are `[u64; 24]` (Fp2 x ++ Fp2 y, 6 LE limbs per Fp element).
GT elements (Fp12) are `[u64; 72]`.
:::

:::info Hints feature gate
Most signatures take a final `#[cfg(feature = "hints")] hints: &mut Vec<u64>`
parameter. The argument is only present when the crate is built with the
`hints` Cargo feature; without it the parameter (and the corresponding
call-site argument) is removed by `cfg`. Pass a mutable `Vec<u64>` collector
to record syscall hints, used by the prover to seed nondeterministic
witness data.
:::

## Base field (Fp)

Arithmetic mod the BLS12-381 base prime `p` (~381 bits, 6 LE
limbs). Fp is the foundation for every other layer on this page:
G1 coordinates, the components of Fp2, and ultimately Fp12 are
all expressed as towers over Fp.

## Extension field (Fp2)

The quadratic extension Fp2 = Fp[u] / (u² + 1). Elements are pairs
(re, im) of Fp values. G2 coordinates are Fp2 elements, and Fp6
and Fp12 are towers built on top of Fp2, so this section feeds
both the G2 twist and the pairing target.

## Scalar field (Fr)

Arithmetic mod the prime-order subgroup order `r`. Fr is the field
of valid scalar multipliers for G1 and G2. Reducing an arbitrary
256-bit integer into Fr is the standard preprocessing step before
calling any of the scalar multiplication routines below.

## Pairing

The bilinear map `e: G1 × G2 → Fp12`. The Miller loop accumulates
line evaluations along a chain of G2 doublings and additions, and
final exponentiation lifts the result into the cyclotomic
subgroup `GΦ₆(p²)` so pairing equality is well-defined. KZG
verification and BLS signature verification both reduce to a
small fixed-size pairing check.

## KZG

The KZG polynomial commitment scheme built on the BLS12-381
pairing. A commitment `C` to a polynomial `f` and an opening proof
`π` at point `z` to value `y` together satisfy
`e(C - [y]G₁, G₂) = e(π, [τ]G₂ - [z]G₂)`. This single pairing
equation is the verifier the Ethereum blob market (EIP-4844) uses,
so this section is the reason most guests will reach for
BLS12-381 instead of BN254.

<FnPanel>

<FnDef name="verify_kzg_proof">

Verifies a KZG opening proof: checks
e(C − [y]G₁, G₂) = e(π, [τ]G₂ − [z]G₂). All inputs are
big-endian byte arrays.

```rust
pub fn verify_kzg_proof(
    z_bytes: &[u8; 32],
    y_bytes: &[u8; 32],
    commitment_bytes: &[u8; 48],
    proof_bytes: &[u8; 48],
    #[cfg(feature = "hints")] hints: &mut Vec<u64>,
) -> bool
```

#### Parameters

| Name | Type | Description |
|---|---|---|
| `z_bytes` | `&[u8; 32]` | Evaluation point as a big-endian 32-byte scalar. |
| `y_bytes` | `&[u8; 32]` | Claimed evaluation value as a big-endian 32-byte scalar. |
| `commitment_bytes` | `&[u8; 48]` | KZG commitment C as a compressed G1 point (big-endian). |
| `proof_bytes` | `&[u8; 48]` | Opening proof π as a compressed G1 point (big-endian). |
| `hints` | `&mut Vec<u64>` | Hint collector (only present under the `hints` feature). |

#### Returns

| Type | Description |
|---|---|
| `bool` | `true` if the KZG proof is valid, `false` otherwise (including parsing or non-canonical scalar errors). |

#### Example
