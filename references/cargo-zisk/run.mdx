Run the guest program inside the ZisK emulator. No proof is generated
and no execution trace is produced. Use `run` for fast iteration:
verify that your logic is correct and profile execution cost before
committing to the full proving pipeline.

:::tip run vs execute
`run` uses a lightweight emulator. It is the fastest way to check
guest logic and profile cycle cost. It does not replicate the full
proving pipeline. Use [`execute`](/references/cargo-zisk/execute)
when you need to validate that your inputs and hint streams are
correct against the same execution path that `prove` uses.
:::

```bash
cargo-zisk run [-i <input-file>] [-e <elf>] [-p <mode>]
```

## Flags

| Flag                    | Short | Default               | Description                                                                                    |
|-------------------------|-------|-----------------------|------------------------------------------------------------------------------------------------|
| `--features`            | `-F`  | None                  | Space or comma separated list of features to activate.                                         |
| `--all-features`        |       | `false`               | Activate all available features.                                                               |
| `--no-default-features` |       | `false`               | Do not activate the `default` feature.                                                         |
| `--release`             |       | `false`               | Build artifacts in release mode, with optimizations.                                           |
| `--elf <elf>`           | `-e`  | standard release path | Path to the guest ELF generated during build.                                                  |
| `--inputs <input-file>` | `-i`  | none                  | Path to a `ZiskStdin`-formatted input file.                                                    |
| `--profiling <mode>`    | `-p`  | none (no profiling)   | Profiling report to emit. Options: `inline`, `summary`, `complete`. Omitted means no profiling. |

## Examples

Run the guest with no input:

```bash
cargo-zisk run
```

Run with a binary input file:

```bash
cargo-zisk run --inputs ./inputs/data.bin
```

Print the full profiling report to stdout:

```bash
cargo-zisk run --inputs ./inputs/data.bin --profiling summary
```

Save the full profiling report to disk for interactive exploration:

```bash
cargo-zisk run --inputs ./inputs/data.bin --profiling complete
```

## Outputs

When `--profiling` is omitted, no profiling output is produced. Pass one of the three modes below to enable a report.

**`inline`**: Prints the accumulated cost of each named region defined with `profile_start!` / `profile_end!` markers in the guest code:

```
[`inline tag1`]    320,750
[`inline tag2`]    823,540
```

**`complete`**: writes the full profiling report to `profiling/`
in the workspace root. Load the file at
[profiler.firefox.com](https://profiler.firefox.com) to explore the
call stack and inverted call stack interactively.
