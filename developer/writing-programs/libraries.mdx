<Summary>

Two paths route expensive operations through ZisK native precompiles. Patches accelerate a whole dependency with one `Cargo.toml` change; `zisklib` wrappers replace call sites when no patch exists.

</Summary>

## Why this matters

ZisK exposes native operations called **precompiles** that execute cryptographic and arithmetic primitives in a single VM step instead of unrolling them into thousands of base-ISA instructions.

Without a precompile, a SHA-256 hash compresses to a few hundred bytes of input through more than a thousand RISC-V instructions per block, and every one of those instructions costs cycles in the proof.

With a precompile, the same hash dispatches to the SHA-256 chip in one step. The cost reduction is typically two to three orders of magnitude on hash-heavy or curve-heavy guests.

Patches and `zisklib` wrappers are two ergonomic layers on top of the underlying precompiles.

- **Patches** swap a crate dependency in `Cargo.toml` with a ZisK-accelerated fork. Every crate in the workspace that depends on the patched library is accelerated automatically, with no changes to any source file. The trade-off is version pinning: each patch is tied to a specific upstream version.

- **Precompile wrappers** let you replace individual call sites with safe Rust functions that dispatch directly to the corresponding ZisK state machine. No `unsafe`, no manual operand layout, no version pinning. The trade-off is that each call site has to be updated by hand.

Pick the patch path first when a matching fork exists; fall back to wrappers when it does not, or when you want the absolute lowest cost.

## Applying a patch

This guide uses the same `merkle-tree` project from the [profiling guide](./profiling). If you completed that guide, the project is already on disk; skip ahead to [Getting a baseline](#getting-a-baseline).

If you are starting fresh, scaffold the project first:

```zisk-bash title="bash"
cargo-zisk new merkle-tree
cd merkle-tree
```

The template lays out a `guest/` crate where the ZisK program lives and a `common/` crate with shared types and logic.

Open `common/src/lib.rs` and write the shared Merkle-tree logic:

```rust title="common/src/lib.rs"
pub use sha2::{Digest, Sha256};

pub type Hash = [u8; 32];

pub fn merkle_root(mut leaves: Vec<Hash>) -> Hash {

    while leaves.len() > 1 {
        leaves = leaves
            .chunks(2)
            .map(|p| {
                let mut h = Sha256::new();
                h.update(p[0]);
                h.update(p.get(1).unwrap_or(&p[0]));
                h.finalize().into()
            })
            .collect();
    }
    leaves[0]
}
```

Now open `guest/src/main.rs` and write the program:

```rust title="guest/src/main.rs"
#![no_main]
ziskos::entrypoint!(main);

use common::{merkle_root, Hash, Digest, Sha256};
use ziskos::io;

fn main() {

    let n: u64 = 1024;

    let leaves: Vec<Hash> = (1..=n)
        .map(|i| Sha256::digest(i.to_le_bytes()).into())
        .collect();

    let root = merkle_root(leaves);

    io::commit_slice(&root);

    println!("Merkle root for {} leaves: {}", n, hex::encode(root));

}
```

### Adding the patch

The guest code stays exactly as it was. The patch entry goes in the workspace `Cargo.toml`:

```toml title="Cargo.toml (workspace root)"
[patch.crates-io]
sha2 = { git = "https://github.com/0xPolygonHermez/zisk-patches", package = "zisk-patch-hashes", tag = "patch-sha2-0.10.9-zisk-0.15.0" }
```

The `tag` field pins the patch to a specific upstream version. The override applies to the whole workspace, so every crate that depends on `sha2` resolves to the accelerated fork, without changes to any source file.

Tags are released alongside ZisK itself; check the [available patches](#available-patches) table below for the current tag for a given crate.

With the patch in place, rebuild and run the same profiling command as the baseline. Cargo resolves `sha2` to the patched fork automatically; nothing else changes.

The guest binary changes because the SHA-256 implementation is different, but the program logic, inputs, and committed output stay identical:

The total cost dropped from 1,804,351,344 to 6,890,240. A reduction of more than 99%, with zero changes to guest code.

`sha2::sha256::compress256`, which previously accounted for 81.2% of the top-cost functions, no longer appears in the table at all.

SHA-256 now executes as a native precompile at 11.4% of total cost, and the xor/or/shift-heavy opcode pattern it produced has vanished entirely.

The cost that remains is dominated by the Merkle tree structure itself: memory accesses, vector allocation, the loop. Those are irreducible regardless of how SHA-256 is implemented.

:::warning Patch version pinning

Each patch is tied to a specific version of the upstream crate via the `tag` field. Upgrading `sha2` in your `Cargo.toml` to a version that has no matching patched fork makes the patch entry fail to resolve, and the build errors out.

Before upgrading any patched dependency, check the [available patches](#available-patches) table to confirm a fork exists for the target version.

If it does not, either stay on the current version or replace the call sites with `zisklib` wrappers, which are version-independent.

:::

## Applying a precompile wrapper

Reach for `zisklib` when no patch exists, to avoid version pinning, or for maximum efficiency. It calls the ZisK precompile directly with no intermediate crate layer, producing fewer guest instructions than a patch.

`zisklib` is a regular dependency, so it must be declared in every crate that uses it. In this example that means both `common` and `guest`.

### Updating the common crate

Continuing with the same `merkle-tree` project, replace `sha2` with `zisklib` in the common crate dependencies:

```toml title="common/Cargo.toml"
[dependencies]
zisklib = { workspace = true }
```

Then update `common/src/lib.rs` to replace the `sha2` calls with `zisklib::sha256`:

```rust title="common/src/lib.rs"
pub type Hash = [u8; 32];

pub fn merkle_root(mut leaves: Vec<Hash>) -> Hash {

    while leaves.len() > 1 {
        leaves = leaves
            .chunks(2)
            .map(|p| {
                let mut buf = [0u8; 64];
                buf[..32].copy_from_slice(&p[0]);
                buf[32..].copy_from_slice(p.get(1).unwrap_or(&p[0]));
                zisklib::sha256(&buf)
            })
            .collect();
    }
    leaves[0]
}
```

### Updating the guest crate

The guest crate needs the same swap. Replace `sha2` with `zisklib` in `guest/Cargo.toml`:

```toml title="guest/Cargo.toml"
[dependencies]
ziskos  = { workspace = true }
zisklib = { workspace = true }
common  = { path = "../common" }
```

Then update `guest/src/main.rs` to drop the `Sha256` import and replace the leaf-generation call with `zisklib::sha256`. The rest of the program stays the same.

```rust title="guest/src/main.rs"
#![no_main]
ziskos::entrypoint!(main);

use common::{merkle_root, Hash};
use zisklib;
use ziskos::io;

fn main() {

    let n: u64 = 1024;

    let leaves: Vec<Hash> = (1..=n)
        .map(|i| zisklib::sha256(&i.to_le_bytes()))
        .collect();

    let root = merkle_root(leaves);

    io::commit_slice(&root);

    println!("Merkle root for {} leaves: {}", n, hex::encode(root));

}
```

## Available wrappers

For the full list of `zisklib` function signatures grouped by curve and operation, see the [zisklib reference](/references/zisk-os/zisklib).

The wrappers cover the same primitive families as the patches: hash functions, secp256k1 / secp256r1 ECDSA, BN254 and BLS12-381 pairings, modular exponentiation, and KZG verification.

---

## Summary

Patches and precompile wrappers both route expensive operations through ZisK native precompiles and produce large cost reductions on hash- and curve-heavy guests.

Patches require no guest code changes but are version-pinned to a specific upstream release. Wrappers require updating each call site but carry no version constraint and stay current as ZisK evolves.

Check the [patches table](#available-patches) first: if a fork exists, one line in `Cargo.toml` is all it takes. If it does not, swap to `zisklib` wrappers at the call sites that dominate cost.
