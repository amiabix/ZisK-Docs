## Why profiling matters

Proving is expensive. A program that runs in milliseconds natively can take minutes to prove if it was not written with the zkVM in mind.

Profiling shows where the cost goes before you commit to a full proving run. The earlier in development you measure, the cheaper it is to fix.

:::warning **Profiling cost** and **final cost** are not the same thing

Optimizing by profiling cost does not guarantee a proportional
reduction in proving cost, but it is strongly recommended before
moving to real costs.

- **Profiling cost** are the direct operational costs of your code,
calculated using the optimal cost per operation. They scale
proportionally with your changes, making them the right signal
during optimization: every improvement is immediately reflected in
the numbers.

- **Proving cost** depend on how the prover allocates resources at runtime
and cannot be predicted from profiling alone. A single extra
operation can push past an instance boundary and double that
component's cost. Use profiling costs to optimize, and real costs
to size your proving run.

:::

## Profiling a guest program

This section profiles a guest that hashes a fixed `N` indices into SHA-256 leaves and reduces them into a Merkle root. You apply the three profiling tools in sequence; each one builds on what the previous one revealed.

### Creating the project

Scaffold the project with `cargo-zisk`. The CLI clones the standard ZisK template and wires the workspace, toolchain, and dependencies in one step:

```zisk-bash title="bash"
cargo-zisk new merkle-tree
cd merkle-tree
```

The template lays out a `guest/` crate where the ZisK program lives and a `common/` crate for shared types and logic.

Open `common/src/lib.rs` and write the shared Merkle-tree logic:

```rust title="common/src/lib.rs"
pub use sha2::{Digest, Sha256};

pub type Hash = [u8; 32];

pub fn merkle_root(mut leaves: Vec<Hash>) -> Hash {

    while leaves.len() > 1 {
        leaves = leaves
            .chunks(2)
            .map(|p| {
                let mut h = Sha256::new();
                h.update(p[0]);
                h.update(p.get(1).unwrap_or(&p[0]));
                h.finalize().into()
            })
            .collect();
    }
    leaves[0]
}
```

Open `guest/src/main.rs` and write the program. It hashes each leaf index with SHA-256 to produce `N` leaves, then reduces them into a single Merkle root:

```rust title="guest/src/main.rs"
#![no_main]
ziskos::entrypoint!(main);

use common::{merkle_root, Hash, Digest, Sha256};
use ziskos::io;

fn main() {

    let n: u64 = 1024;

    let leaves: Vec<Hash> = (1..=n)
        .map(|i| Sha256::digest(i.to_le_bytes()).into())
        .collect();

    let root = merkle_root(leaves);

    io::commit_slice(&root);

    println!("Merkle root for {} leaves: {}", n, hex::encode(root));

}
```

Build and run it once to confirm the logic is correct before profiling:

```zisk-bash title="bash"
cargo-zisk build --release
cargo-zisk run
```

### Using inline cost markers

Before reaching for heavier tools, wrap the regions of interest with `profile_report_start!` and `profile_report_end!` markers.

The macros are exported from `ziskos::profile`. They are re-exported at the crate root, so a guest with `use ziskos::*;` already has them available; otherwise import them directly with `use ziskos::profile::*;`.

When the program exits, the cost of each named region is printed to stdout. That single table is often enough to point at the section that deserves a closer look.

Open `guest/src/main.rs` and wrap the two phases of the program:

```rust title="guest/src/main.rs"
#![no_main]
ziskos::entrypoint!(main);

use common::{merkle_root, Hash};
use sha2::{Digest, Sha256};
use ziskos::io;

fn main() {

    let n: u64 = 1024;

    profile_report_start!(PREPARE_LEAVES);
    let leaves: Vec<Hash> = (1..=n as u64)
        .map(|i| Sha256::digest(i.to_le_bytes()).into())
        .collect();
    profile_report_end!(PREPARE_LEAVES);

    profile_report_start!(COMPUTE_ROOT);
    let root = merkle_root(leaves);
    profile_report_end!(COMPUTE_ROOT);

    io::commit_slice(&root);

    println!("Merkle root for {} leaves: {}", n, hex::encode(root));

}
```

Rebuild and run:

```zisk-bash title="bash"
cargo-zisk build --release
cargo-zisk run --release
```

#### Report summary

The first box is a global snapshot of the execution:

- **STEPS**: number of processor cycles executed. Proportional to how long the
  program runs; doubling N should roughly double the step count.
- **COST**: total profiling cost expressed in proof area units. This is the metric
  to minimize: higher cost means longer proof generation.
- **RAM**: heap memory used vs. total available. Only reported when using ZisK's
  default bump allocator (which never frees memory and avoids heap-management overhead).

#### Cost distribution summary

The second box breaks cost into five categories:

| Category      | What it measures  |
|---------------|-------------------|
| **Base**      | Fixed overhead such as tables, range checks, and other constant components that exist regardless of program logic. |
| **Main**      | Processor execution cost, directly proportional to STEPS. Reducing STEPS reduces Main. |
| **Opcodes**   | Simple 64-bit operations (`a op b = c, flag`): arithmetic, bitwise, shifts. |
| **Precompiles** | Complex operations whose parameters exceed 64 bits and use memory as an exchange channel |
| **Memory**    | Direct memory reads/writes and the extra state machines needed for non-aligned or sub-8-byte accesses. |

#### Cost by opcode summary

The third box has two panels side by side. The left lists every opcode executed with its total cost and share.

The right panel shows **FROPs** (Frequently-used OPerationS), common low-level patterns the emulator detects and pre-calculates.

The **HIT %** column tells you how often each pattern matched. The FROPS total in the summary box is the cost that would be added if these optimizations did not exist; the savings are already baked into the numbers you see.

#### Top cost functions

The fourth box ranks guest functions by **cumulative cost**. Each entry includes the cost of every nested call it makes, not just its own instructions.

This is the right lens for deciding where to optimize: a function with 60% cumulative cost is the root cause, even if its own instructions are cheap.

#### Summarized report conclusions

Three things stand out:

- **Precompiles are at 0.0%.** SHA-256 runs entirely as plain opcodes, with no native acceleration.
- **`sha2::compress256` tops the cost functions** at 81.2%. The hash computation dominates total cost.
- **The opcode table is dominated by SHA-256 operations.** `xor`, `or`, `srl_w`, `sll`, and `and` together account for the bulk of opcode cost, all characteristic of the SHA-256 round function.

The conclusion is already clear: SHA-256 needs to be replaced with a precompile-backed implementation. The next page covers exactly that.

If you want to understand the full call structure first, which paths reach `compress256` and from where, the call-stack view shows it in depth.

### Exploring the call stack

The summarized report named `sha2::compress256` as the dominant cost driver, but it cannot show how execution reaches it, or whether heap allocation is compounding the pressure.

The call-stack view traces exactly how execution flows into each hot spot. Generate it with:

```zisk-bash title="bash"
cargo-zisk run --profiling complete
```

This writes `complete-report.json.gz` to the current directory. Open it at [profiler.firefox.com](https://profiler.firefox.com) via **Load a profile from file** and select the file. Firefox is recommended for large profiles.

- **Call stack (top-down)** lets you understand cost in terms of
  entry calls: starting from `main`, you can follow exactly which
  calls are responsible for the cost at each level. The flamegraph
  visualises this as nested bars, each proportional to the cost
  accumulated through that entry path.

- **Inverted call stack (bottom-up)** flips the perspective: instead
  of following calls down, it ranks leaf functions by cost.
  `sha2::compress256` surface immediately at the top as it is called in
  both phases we've profiled.

The call stack leaves no ambiguity: `compress256` is the single bottleneck, reached through both the leaf-hashing and tree-reduction phases. The next step is replacing it with a native precompile.
