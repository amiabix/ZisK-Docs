# What is ZisK

ZisK is a STARK-based zero-knowledge virtual machine (zkVM)
that executes programs compiled from standard instruction set
architectures and produces succinct, transparent proofs of
their correct execution.

ZisK targets the RISC-V ISA, allowing developers to keep a
familiar Rust/C programming model while building through the ZisK
toolchain.

ZisK's modular architecture enables the addition of optimised
constraint systems for expensive operations such as
elliptic-curve arithmetic and bulk memory access, making it
practical to prove complex computations at scale.

## What ZisK unlocks

Recent advances in zero-knowledge proof systems opened the
door to applications that verify the correct execution of an
arbitrary computation, without revealing private inputs and in
time and space sublinear in the original computation.

ZisK is built to make those applications practical at
production scale. Concretely:

- **Validity rollups:** Layer-2 chains posting a single proof
  of millions of transactions instead of replaying every
  transaction on Layer-1.
- **Verifiable coprocessors:** off-chain workloads such as
  large-state reads or complex business logic, whose output a
  smart contract can trust without re-running them.
- **Trustless cross-chain bridges:** chain A verifying chain
  B's state without trusting an oracle.
- **Off-chain computation protocols:** services that publish
  proofs of their outputs alongside the outputs themselves, so
  third parties can confirm the computation happened correctly
  without re-executing it.
- **Privacy-preserving computation:** proving a computation
  over private data without revealing the data.

## Properties of a ZisK proof

Four properties anchor everything else in this section. The
rest of the docs uses them as shorthand:

- **Soundness:** a verifier cannot be tricked into accepting a
  proof of a computation that did not actually run that way.
  The prover cannot fake a result without breaking a hardness
  assumption.
- **Completeness:** every honest execution can be proved. If
  the guest ran correctly, the prover can always produce a
  proof a verifier will accept.
- **Succinctness:** the proof and the verification cost are
  small and roughly independent of the original computation's
  size. A million-step proof verifies in about the same time
  as a thousand-step proof.
- **Zero-knowledge:** the proof reveals nothing about the
  inputs beyond what the guest explicitly committed as public
  outputs. Private inputs stay private even though their
  effect on the outputs is verifiable.

These properties come from the underlying STARK proof system.
ZisK does not add or weaken any of them. The design choices
in the rest of this section govern how fast and how cheaply a
proof with these properties can be produced.

## Trust model

A complete picture of "verifying a ZisK proof" is more nuanced
than calling `verify()`. Three things must hold:

1. **The proof is valid:** this is what `proof.verify()` checks.
   It guarantees the proof is well-formed and consistent with
   the program verification key embedded in the proof.
2. **The guest program is the one you expected:** the
   **program verification key** is a small commitment derived
   from the program ROM during setup and included with the proof.
   The verifier can override or compare it against an audited
   value to confirm the proof was produced for the expected guest.
   If you do not control or audit the source the binary was built
   from, you cannot know what the guest actually does.
3. **The committed public values match what your application
   requires:** a valid proof of an irrelevant computation is
   still useless. Read the public values back and assert they
   match what you expected.

A malicious prover could otherwise submit a proof generated by
a trivial guest that simply commits whatever it wants. The
proof would verify yet attest to nothing meaningful.

What you *do* trust:

- **Cryptographic assumptions:** STARKs and LtHash rely on
  collision-resistant hashing and the Short Integer Solution
  lattice problem.
- **Compiler and toolchain:** the toolchain that produced the
  ELF. You can audit it yourself.
- **Guest source publisher:** the party that published the
  guest source. You can audit that yourself.

What you *do not* trust:

- **The prover:** the prover can be malicious, run on
  compromised hardware, or see your private inputs. None of
  that affects soundness.
- **The verifier host:** verification is deterministic. If it
  returns `Ok`, the proof is valid no matter who ran the
  verifier.
